## 1. PM/PL이란? (크라우드소싱/데이터 구축 회사 맥락)

### 1.1 일반적 PM vs PL 차이

| 구분 | PM (Project Manager) | PL (Project Leader) |
|------|---------------------|---------------------|
| **초점** | 프로젝트 전체 관리 (일정/비용/품질/리스크) | 기술적 리더십 + 실무 관리 |
| **역할** | 고객사 소통, 전체 일정 조율, 보고 | 작업 기준 설정, 품질 검수, 작업자 관리 |
| **보고** | 고객사 + 내부 경영진 | PM에게 보고 또는 PM과 동급 |
| **기술** | 관리 역량 > 기술 역량 | 기술 역량 >= 관리 역량 |

### 1.2 데이터 구축 회사에서의 PM/PL

**크라우드웍스 같은 데이터 구축 회사에서의 PM/PL은 일반 IT 회사의 PM과 다르다.**

일반 IT PM: 소프트웨어 개발 프로젝트 관리 (개발자 -> 코드 -> 배포)
데이터 PM/PL: **크라우드소싱 데이터 작업 관리** (작업자 -> 데이터 -> 납품)

핵심 차이:
- **관리 대상**: 개발자 5~20명이 아닌, 크라우드 작업자 수십~수백명
- **산출물**: 코드가 아닌 데이터 (JSONL, 이미지 라벨 등)
- **품질 관리**: 코드 리뷰가 아닌 데이터 검수 (정확성, 일관성, 완결성)
- **커뮤니케이션**: 비전문 작업자 교육/안내가 핵심

### 1.3 크라우드웍스 수리논술 PM/PL 구체적 역할

채용 공고 원문 기반 정리:

```
[담당업무 3대 축]

1. 수식·데이터 산출물 품질 관리 (핵심)
   - LaTeX 구문 기반 수식 데이터 모니터링
   - 고객사 요구사항 반영한 산출물 검수 및 데이터 정합성 점검
   - JSONL 등 데이터 포맷 구조 확인 및 오류 검토

2. 프로젝트 운영 및 커뮤니케이션
   - 작업자 진척률 및 참여도 관리
   - 작업자-검수자-PM 간 중간 소통
   - 작업 기준 및 운영 정책 안내

3. 작업자·검수자 교육 및 운영 지원
   - 온라인 교육 진행
   - 프로젝트 이해 및 작업 툴 사용 교육 운영
```

**해석:** 이 포지션은 PM과 PL의 중간 -- "프로젝트 운영/관리"와 "기술적 품질 관리"를 모두 담당. 소규모 프로젝트(PM/PL 1명)이므로 한 사람이 두 역할을 겸임.

## 2. 재택 PM/PL의 일상 (Daily Routine 추정)

### 2.1 하루 일과 시뮬레이션

```
09:00 - 09:30  [출근] 메신저/이메일 확인, 전날 작업 현황 체크
09:30 - 10:30  [데이터 검수] 전날 제출된 데이터 샘플링 검수
               - JSONL 구조 확인 (자동 검증 스크립트 결과 확인)
               - LaTeX 수식 정확성 스팟 체크
               - 오류 발생 작업자 목록 정리
10:30 - 11:00  [커뮤니케이션] 작업자 문의사항 대응 (카카오톡/슬랙)
               - "이 수식 LaTeX으로 어떻게 쓰나요?"
               - "검수 반려 사유가 뭔가요?"
11:00 - 12:00  [모니터링] 진척률 대시보드 업데이트
               - 작업자별 완료량/불량률 확인
               - 일정 대비 진행률 체크
               - 리스크 시그널 탐지

12:00 - 13:00  점심

13:00 - 14:00  [고객사 소통] (필요 시)
               - 고객사 요구사항 변경 대응
               - 중간 리포트 제출
               - 데이터 스키마/품질 기준 협의
14:00 - 15:30  [교육/안내] (주 1-2회)
               - 신규 작업자 온보딩 교육 (온라인)
               - 자주 발생하는 오류 패턴 공지
               - 작업 가이드라인 업데이트
15:30 - 16:30  [품질 관리]
               - 오류 패턴 분석 (어떤 유형의 실수가 반복되는지)
               - 자동 검증 규칙 보완 요청
               - 검수자 피드백 수합
16:30 - 17:00  [일일 마감]
               - 당일 진행 현황 정리
               - 내일 우선순위 설정
               - 팀/고객사 보고 (필요 시)
```

### 2.2 주간/월간 반복 업무

**주간:**
- 월: 주간 작업 계획 수립, 목표 배분
- 금: 주간 실적 리포트 작성, 다음 주 계획

**월간:**
- 월초: 월간 목표 설정, 작업자 인원 조정
- 월말: 월간 실적 리포트, 고객사 중간 보고, 품질 트렌드 분석

### 2.3 재택근무 특수성

**장점:**
- 집중 가능 (데이터 검수는 집중력 필요)
- 작업자 대부분도 재택이므로 온라인 소통이 자연스러움
- 시간 유연성 (작업자 마감 관리가 핵심이지 출퇴근이 아님)

**도전:**
- 비대면 커뮤니케이션의 한계 (뉘앙스 전달 어려움)
- 자기 관리 (일-생활 경계 모호)
- 고객사 미팅은 온라인이지만, 긴급 상황 시 대면 가능성

## 4. 품질 관리 (QA/QC) 워크플로

### 4.1 크라우드소싱 데이터 품질 관리 체계

```
[작업자]  →  [1차 자동 검증]  →  [검수자]  →  [PM/PL 최종 확인]  →  [납품]
  작업         스크립트          사람 검수       샘플링 검수         고객사
```

**1단계: 작업자 작업**
- 작업 가이드라인에 따라 데이터 생성 (수리논술 문제 -> LaTeX 변환)
- Workstage Cloud 플랫폼에서 직접 작업

**2단계: 자동 검증 (스크립트)**
- JSONL 구조 파싱 (필수 필드 존재, 타입 일치)
- LaTeX 구문 검사 (괄호 균형, 기본 명령어 유효성)
- 중복 검사 (ID 유니크, 내용 유사도)

**3단계: 검수자 수동 검수**
- 수학적 정확성 확인 (풀이가 맞는지)
- LaTeX 렌더링 확인 (수식이 의도대로 보이는지)
- 논리적 완결성 (풀이 과정에 논리적 빈틈 없는지)

**4단계: PM/PL 최종 확인**
- 전수 검수가 아닌 **샘플링 검수** (전체의 5~10%)
- 오류율 통계 기반 판단 (목표: 99% 이하 오류율)
- 반복 오류 패턴 식별 -> 가이드라인 보완

### 4.2 크라우드웍스의 품질 보증

크라우드웍스는 **"100% 전수 검수"**를 통해 **데이터 품질 99%**를 보장한다고 공식 발표. 이는 PM/PL이 직접 전수 검수하는 것이 아니라, 검수자 + 자동화 + PM 샘플링의 다층 구조로 달성.

### 4.3 품질 관리 KPI (추정)

| KPI | 설명 | 목표치 (추정) |
|-----|------|-------------|
| 오류율 | 전체 데이터 중 오류 비율 | < 1% |
| 납기 준수율 | 일정 대비 완료율 | >= 95% |
| 작업자 참여율 | 배정 작업자 중 실제 활동 비율 | >= 80% |
| 재작업률 | 반려 후 재작업 비율 | < 5% |
| 검수 소요시간 | 건당 평균 검수 시간 | 모니터링 |

## 5. 필수 vs 우대 스킬 분석

### 5.1 진짜 필수 스킬 (없으면 업무 불가)

| 스킬 | 이유 | 수준 |
|------|------|------|
| **프로젝트 관리 역량** | 일정/인력/품질을 동시에 관리해야 함 | 실무 경험 필수 |
| **커뮤니케이션** | 작업자(비전문가) + 검수자 + 고객사와 모두 소통 | 상 |
| **Excel/Google Sheets** | 진척률 관리, 통계, 리포트가 매일 발생 | 중상 |
| **수학 기본 소양** | 수리논술 데이터의 정확성을 판단할 수 있어야 함 | 중 (고교 수학 이상) |
| **꼼꼼함 / 데이터 감각** | 오류 탐지, 패턴 파악이 핵심 업무 | 상 |

### 5.2 중요하지만 학습 가능한 스킬

| 스킬 | 이유 | 학습 난이도 |
|------|------|------------|
| **LaTeX 기본 문법** | 수식 모니터링 필요, 하지만 직접 작성보다 "읽기/검수" 수준 | 낮음 (1-2주) |
| **JSONL 이해** | 구조 이해 + 오류 식별 수준 | 낮음 (1주) |
| **크라우드소싱 플랫폼 운영** | Workstage 사용법은 입사 후 교육 가능 | 중 |

### 5.3 있으면 좋은 스킬 (우대사항)

| 스킬 | 이유 | 임팩트 |
|------|------|--------|
| **수학교육과/수학과 전공** | 수리논술 내용 이해도가 높으면 검수 품질 향상 | 높음 |
| **Python 기초** | 자동 검증 스크립트 이해/활용 가능 | 중 |
| **데이터 구축/검수 경험** | 크라우드소싱 워크플로 이해가 빠름 | 높음 |
| **데이터 포맷 경험** | JSONL, CSV, 스키마 설계 경험 | 중 |

### 5.4 면접에서 어필할 포인트 (전략적 분석)

**이 포지션의 진짜 핵심은 "품질 관리 + 사람 관리"이다.**

LaTeX이나 JSONL은 학습 가능한 기술 스킬이지만, 다음은 쉽게 대체 불가:
1. **다수의 비전문 작업자를 원격으로 관리하는 능력**
2. **데이터 오류 패턴을 빠르게 파악하는 분석력**
3. **고객사 요구사항과 실제 작업 가능 범위 사이를 조율하는 소통 능력**
4. **일정 지연이나 품질 하락 시 신속한 대응 능력**

## 7. 고용 형태 분석 (Critical Insight)

### 7.1 정규직 vs 프리랜서/계약직 가능성

**이 포지션이 프로젝트 기반 계약일 가능성이 높은 이유:**

1. **"재택" 강조**: 크라우드웍스 정규직은 기본 출퇴근, 재택은 팀 협의 후 가능. "재택" PM/PL은 프로젝트 단위일 가능성.
2. **"1명" 모집**: 특정 프로젝트에 특화된 인력.
3. **크라우드잡스 사업 모델**: 크라우드웍스 자체가 긱워커 매칭 플랫폼을 운영. PM/PL도 이 모델로 운영 가능.
4. **수리논술이라는 특정 도메인**: 범용이 아닌 특정 프로젝트 한정.

### 7.2 면접 시 확인 사항

반드시 확인해야 할 질문:
1. **고용 형태**: 정규직? 계약직? 프리랜서?
2. **프로젝트 기간**: 단기(3~6개월)? 장기(1년+)?
3. **팀 구성**: 함께 일하는 검수자, 다른 PM이 있는지?
4. **고객사**: 누구를 위한 데이터인지? (NDA 한도 내에서)
5. **작업자 규모**: 몇 명의 작업자를 관리하게 되는지?
6. **도구**: Workstage Cloud 외에 어떤 도구를 쓰는지?

## 8. 핵심 인사이트 (Key Insights)

### 8.1 포지션의 본질

> **이 포지션은 "기술 PM"이 아니라 "데이터 품질 관리자 + 크라우드 운영자"이다.**

코딩을 하는 것이 아니고, LaTeX/JSONL은 "도구"일 뿐. 진짜 업무는:
- 수십~수백명의 원격 작업자가 일관된 품질로 데이터를 생산하도록 관리하는 것
- 고객사의 요구사항을 작업자가 이해할 수 있는 가이드라인으로 변환하는 것
- 데이터 품질을 수치로 측정하고 문제를 조기에 발견하는 것

### 8.2 강점으로 어필할 요소

1. **분석력**: 데이터 패턴 파악, 오류 탐지 능력 -> Excel/Python 활용 사례
2. **교육 능력**: 비전문가에게 기술 개념을 설명할 수 있는 역량
3. **자기 관리**: 재택근무에서의 자율성과 책임감
4. **수학 소양**: 수리논술 내용을 이해하고 검수할 수 있는 기본기
5. **기술 호기심**: LaTeX, JSONL을 빠르게 학습할 의지와 능력

### 8.3 주의 사항

1. **회사 재무 상황**: 매출 급감(-49.9%), 적자 지속, 인력 감소. 안정성 우려.
2. **고용 형태 불확실**: 프로젝트 단위 계약 가능성 -> 장기 커리어 계획과 맞는지 확인.
3. **연봉 수준**: 업계 평균 대비 높지 않음 (평균 ~4,000만원).
4. **업무 강도**: 크라우드소싱 PM은 작업자 관리 스트레스가 높을 수 있음 (납기 압박 + 품질 유지).
