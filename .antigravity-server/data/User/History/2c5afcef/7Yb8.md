# Implementation: HWPX Automation & Security

Effective HWP automation requires handling application-specific behaviors and security restrictions.

## 1. Security Bypass Protocol
Hancom Office typically displays "Access Warning" dialogs when controlled via OLE. The following methods are used to achieve unattended execution:

### A. Initialization
The `Hwp` class must be initialized with `register_module=True`. This allows the application to load authorized security modules (like `FilePathCheckerModuleExample.dll`) or registry-based white-lists.
```python
hwp = Hwp(visible=True, register_module=True)
```

### B. Registry Requirements
The security level in HWP must be set to **Low** (낮음) via the user interface or registry modification to allow programmatic file access.

## 2. Path Resolution
Since the pipeline spans two filesystems:
- **Relative Path**: `/home/palantir/hwpx/`
- **Windows Access**: `\\wsl.localhost\Ubuntu\home\palantir\hwpx\`
- **Tooling**: `wslpath -w` is used by `WSLBridge` to generate the correct Windows-side string for the executor.

## 3. Specific HWP Action Patterns

### Multi-Column Layouts
HWP 2024 requires specific `ParameterSet` flags for column definitions.
- **Action**: `"MultiColumn"`
- **Type**: `1` (Newspaper style). This differentiates it from "Parallel" columns which behave more like side-by-side text boxes.
- **Break**: `hwp.HAction.Run("BreakColumn")` is used to force text flow to the next column.

### Table Navigation
- **Mechanism**: `TableCellBlock`.
- **Logic**: Precise navigation to specific cells is required before inserting elements like images or nested text, as HWP's cursor model is document-flow dependent.
